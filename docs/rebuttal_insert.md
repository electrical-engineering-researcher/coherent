# Suggested Rebuttal Insert for Q3/Q4

We agree that the original submission did not provide enough implementation detail for full reproduction. In the revision and supplementary artifact, we will provide the complete reproducibility package containing prompt templates, schemas, scoring scripts, kernel metadata format, generated outputs, and validation logs.

For S3, each candidate interpretation is represented as a structured design object containing module hierarchy, ports, internal signals, FSM states, datapath components, timing assumptions, reset policy, interface definitions, constraints, assumptions, and confidence. This object is serialized into a normalized textual form and encoded using Sentence-BERT `all-mpnet-base-v2` with 768-dimensional L2-normalized embeddings. Candidate plans are clustered using agglomerative hierarchical clustering with average linkage and cosine distance. The clustering threshold is selected from `{0.50, 0.55, 0.60, 0.65, 0.70, 0.75}` by silhouette-score optimization. The dominant cluster is selected by largest cluster size, with average confidence as a tie-breaker. The final conceptual plan is produced by confidence-based merging over the dominant cluster.

We will also include the explicit knowledge-graph schema. Nodes include ports, registers, counters, FSM states, memories, modules, clocks, resets, outputs, constraints, datapath nodes, and control nodes. Edges include drives, reads, updates, resets, transitions-to, depends-on, connects-to, contains, clocks, and enables. Conflict resolution follows a deterministic priority order: explicit specification, interface consistency, timing consistency, reset policy, hardware-safety default, and majority vote. For example, in an 8-bit serial-to-parallel converter, if candidates disagree on whether `parallel_out` updates continuously or only after 8 received bits, the timing rule selects the latter because the specification requires frame-level output timing.

For S2, each kernel is represented by structured metadata containing category, functional tags, port list, parameter options, timing characteristics, synthesis constraints, verification status, and a natural-language description. Kernel ranking uses the fixed formula `Score(k)=0.60*Simemb+0.25*Simtag+0.15*Simif`, where `Simemb` is cosine similarity between query and kernel embeddings, `Simtag` is normalized tag overlap, and `Simif` is interface compatibility over port names, widths, directions, clock/reset signals, and parameters. We use top-k retrieval with `k=5`. Retrieved kernels are adapted through parameter substitution, bit-width scaling, port renaming, signal type alignment, and interface normalization. If direct composition is impossible, S2 generates minimal glue logic such as width adapters, ready/valid adapters, enable/status wiring, or control/status connections.

For S1, the checker loop first performs static structural checks, then compiles the generated VHDL with GHDL, runs the generated testbench, checks functional behavior, and validates structural properties such as reset behavior, FSM completeness, counter termination, protocol handshakes, and interface widths. When a failure occurs, the exact diagnostic is passed back to the repair module as structured JSON containing error type, tool, message, module, line, expected behavior, observed behavior, time, and raw log. Repair is restricted to the affected module unless an interface-level diagnostic requires a wider change. The repair budget is fixed to three iterations; designs failing after three repair attempts are classified as unsuccessful and excluded from synthesis/PPA evaluation.

For Q4, we will document the provenance of all 157 kernels and explicitly separate them from evaluation tasks. The custom T1-T20 task specifications, reference implementations, generated outputs, and golden testbenches were not used to construct the retrieval library. We will additionally perform exact-match and near-duplicate checks between evaluation designs and kernel entries. If any kernel is found to overlap with a custom task category, that kernel will be removed for that category or the overlap will be disclosed.
